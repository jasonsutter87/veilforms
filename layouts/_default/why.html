The Problem With "Secure" Form Builders

Every form builder claims to be secure. They tout SSL encryption, GDPR compliance, and SOC 2 certifications. But here's what they don't tell you:

Your data arrives at their servers in plaintext. They encrypt it after they receive it. They can read it. Their employees can read it. A breach exposes it. A subpoena compels it.

"Encryption at rest" means they have the keys. "Encryption in transit" means it's decrypted when it arrives. You're trusting them with everything.

We built VeilForms because we believe that's backwards. Your users' data should be encrypted before it leaves their browser. We should never be able to read it. Ever.

Client-first isn't a feature. It's a promise.

Technical Deep Dive

Hybrid Encryption

Each submission generates a unique AES-256 key for data encryption. The AES key is then encrypted with your RSA-2048 public key. Only your private key can unlock it.

Web Crypto API

We use the browser's native Web Crypto API‚Äîno external crypto libraries. Keys never exist in JavaScript memory longer than necessary.

Zero Knowledge Storage

Our servers store encrypted blobs with anonymous IDs. No metadata links submissions to users. A breach exposes nothing useful.

PII Pattern Matching

Regex-based detection for emails, phones, SSNs, credit cards, IP addresses, and more. Configurable: warn, strip, or block.

Multi-Tenant Isolation

Each tenant's data is cryptographically isolated. Tenant A cannot access Tenant B's encrypted blobs, even with server access.

Open Source Core

Our encryption, PII detection, and identity modules are open source. Audit them yourself. Trust, but verify.
